## 4. 这次实际遇到的两个烧录错误

## 4.1 默认 Arduino / esptool 上传流程报错

我先按 Arduino 正常上传流程试了两次：

- `UploadSpeed = 460800`
- `UploadSpeed = 115200`

两次都在 `Uploading stub flasher...` 这里失败，错误一致：

```text
A fatal error occurred: Failed to write to target RAM (result was 0107: Checksum error)
```

这说明：

- ROM Bootloader 能握手
- 但往目标 RAM 里装 stub flasher 时，数据校验就失败了

## 4.2 改用 `--no-stub` 直写 flash 仍失败

因为默认流程卡在 stub，我又改成了 `esptool --no-stub write-flash`，试了：

- `115200`
- `57600`

两次都能继续走到擦除 flash，但一开始真正写 flash 的第一个包就失败：

```text
A fatal error occurred: Failed to write to target flash after seq 0 (result was 0105: The format of the received message is invalid)
```

也就是说，现在的症状非常集中：

- **连接能建立**
- **擦除也能开始**
- **但一进入真正的数据写入，首包就坏掉**

## 5. 当前最像什么问题

结合这次现象，我认为当前最可能的是下面几类问题。

### 5.1 板子在烧录时被外设拖垮或干扰

你现在这块 ESP32-S3 已经连了 7.3 寸墨水屏转接板。

这类大屏在某些板子上，会在以下阶段给主板带来额外压力：

- 供电瞬态
- GPIO 被外设强拉
- SPI 线干扰
- 上电时序影响

尤其是现在的现象不是“完全连不上”，而是“数据一写就坏”，这很像：

- 板子本体能进下载模式
- 但外设连接状态让串口烧录过程不稳定

### 5.2 USB 串口桥或当前链路质量仍不稳定

虽然现在能识别了，但 `Vendor ID 0x1A86` 是 WCH 方案，很多通用开发板会用这类 USB-串口桥。

这类桥本身不是不能用，但如果叠加：

- 线材一般
- Hub / 口质量一般
- 板子供电波动

就很容易出现这种：

- 可以握手
- 但大块数据传输校验失败

### 5.3 板子下载模式不够稳定

有时自动复位和自动进下载模式并不稳，尤其是通用板。

虽然能连上，但在开始传大块数据时，状态可能切走或失稳。

## 7. 现在最值得立刻做的动作

下面这一步我认为优先级最高，而且最可能直接让烧录通过。

### 方案 A：先断开墨水屏转接板，只留裸 ESP32-S3 烧录

建议你先把下面这些线全部暂时拔掉：

- `BUSY`
- `RST`
- `DC`
- `CS`
- `SCL`
- `SDA`
- `3V3`
- `GND`

也就是：

- **只保留 ESP32-S3 板子单独接 Mac**
- **不要带墨水屏转接板一起烧录**

原因是这一步能最快判断：

- 到底是 ESP32 板子本身烧录不稳
- 还是墨水屏/转接板连接把烧录链路搞坏了

如果断开屏幕后就能烧进去，那问题就基本锁定为：

- 外设干扰
- 供电拖垮
- SPI/控制线影响下载期状态

### 方案 B：手动进下载模式再烧

在“只留裸板”的前提下，再这样操作：

1. 按住 `Boot`
2. 短按 `RST`
3. 松开 `RST`
4. 继续按住 `Boot` 1~2 秒
5. 再松开 `Boot`
6. 然后立刻再试上传

### 方案 C：如果还不行，再换另一根线 / 另一个 USB 口

因为现在已经从“无枚举”变成“可识别但写失败”，这仍然可能是链路质量问题。
