# ESP32直连Mac仍报0107与当前硬件判定

记录时间：2026-04-04 23:32（Asia/Shanghai）

## 本轮新增确认

用户反馈：
- 现在已经是 **直接插电脑接口**。
- USB 线也换过了，使用了另一根线（用户描述为外接显示器配套的线）。
- 结果仍然一样，上传阶段继续报错：
  - `Failed to write to target RAM (result was 0107: Checksum error)`
  - `Uploading stub flasher...`

这意味着：
- 已经基本排除“单纯是某一根 USB 线坏了”的情况。
- 已经基本排除“只是扩展坞/转接坞导致”的情况。
- 当前阻塞依然是 **ESP32-S3 进入下载后，stub flasher 写入 RAM 失败**。

## 本机进一步核查结果

本轮本机检测到的串口：
- `/dev/cu.usbmodem5A470123231`

进一步读取 USB 设备信息后，识别到：
- USB Product Name: `USB Single Serial`
- USB Serial Number: `5A47012323`
- `idVendor = 6790`，即十六进制 `0x1A86`
- `idProduct = 21971`，即十六进制 `0x55D3`

这个 VID/PID 组合指向 **WCH 系列 USB 转串口桥**（常见于 CH34x / CH910x 家族）。

## 当前判断

到这一步，结合前面的全部实验，优先级最高的可能性已经收敛成下面两类：

### 1. 板子自身下载链路不稳定
表现就是：
- 可以识别芯片；
- 也能开始连接；
- 但在 `Uploading stub flasher...` 这里失败；
- 报 `0107 checksum error`。

这通常说明：
- USB 转串口桥不稳定；
- 板载自动下载电路异常；
- 板子供电/时序/硬件质量存在问题；
- 或板子本身就是边缘兼容型号。

### 2. Mac 当前对这颗 WCH 桥的驱动兼容性不理想
虽然系统已经把它识别成串口设备了，但仍可能在高速/写 RAM 阶段不稳定。

## 和墨水屏的关系

当前阶段还**不是墨水屏点亮问题**，而是更前面的步骤：
- 固件还没有成功烧进去；
- 所以现在还不能进入“首次开机 -> 配网 -> 刷图 -> 点亮屏幕”的阶段。

也就是说，当前正确顺序仍然是：
1. 先解决烧录；
2. 再接回墨水屏；
3. 再验证 AP `InkTime-xxxx`；
4. 再进 `http://192.168.4.1/` 配网；
5. 最后看是否首次刷新成功。

## 现在走到哪一步

当前项目推进状态：
- `Arduino IDE` 已安装；
- `ESP32 Arduino Core` 已安装；
- `GxEPD2` 等依赖已安装；
- 工程可以编译通过；
- 串口可以识别到；
- **但烧录仍卡在 stub flasher 上传阶段**。

所以当前准确状态应标记为：
- **卡在“烧录链路故障排查”**。

## 下一步操作顺序（最终收敛版）

### 第一步：确认板子是否还插在面包板上
即使 USB 是直连 Mac，只要开发板本体还插在面包板上，仍可能存在：
- 引脚偏斜接触；
- 某排针与面包板内部短接/虚接；
- 某些下载相关脚位被拉偏。

要求：
- 把 **ESP32-S3 开发板本体完整拔离面包板**；
- 板子周围不要接任何杜邦线；
- 墨水屏排线/转接板也全部不要接；
- 只保留：`ESP32-S3 裸板 + 一根 USB 线 + Mac`。

### 第二步：再次手动进入下载模式后上传
仍按此前步骤：
1. 按住 `Boot`；
2. 短按一下 `RST`；
3. 松开 `RST`；
4. 继续按住 `Boot` 约 2 秒；
5. 松开 `Boot`；
6. 立即在 Arduino IDE 点 `Upload`。

### 第三步：若仍是 0107，则基本判定这块板子不适合继续作为主板推进
如果在“裸板 + 直连 Mac + 更换 USB 线 + 手动下载模式”后仍报同样错误，那么本项目应进入新的结论：
- 当前这块 ESP32-S3 板子的烧录链路高度可疑；
- 继续在它上面耗时间，收益会很低；
- 最务实的路径是：
  - 换一块新的 ESP32-S3 开发板（优先 Espressif 官方或成熟稳定型号）；
  - 或者准备一块已验证可正常烧录的 ESP32-S3 作为替代。

## 本轮给用户的简化结论

一句话版本：

> 现在已经不是线的问题了，也不再是是否直连电脑的问题了；当前最关键的是先把开发板彻底脱离面包板，再做一次“裸板手动下载模式上传”。如果还报 `0107`，就应优先怀疑这块板子的下载链路/USB 桥本身有问题。
