# Mac热点连接界面与命令行IP状态不一致排查

记录时间：2026-04-07 13:23（Asia/Shanghai）

## 本轮现象

用户截图显示：
- Mac 菜单栏 Wi‑Fi 下拉里已经选中了：`mengbj_iPhone`

同时 ESP32 串口也显示：
- `target ssid=mengbj_iPhone`
- `connected`
- `IP=172.20.10.10`

但本轮在这台 Mac 的命令行侧检查时，当前仍只看到：
- `en0 = 10.142.16.119`
- 并且 `networksetup -getairportnetwork en0` 返回：
  - `You are not associated with an AirPort network.`

这说明：
- **UI 显示与命令行看到的网络状态暂时不一致。**

## 当前可确定的事实

### 1. ESP32 侧已经连上了手机热点
这点已经由串口日志确认，无需怀疑：
- `[WIFI] connected`
- `[WIFI] IP=172.20.10.10`

### 2. Mac 本地 `server.py` 仍在正常运行
本轮再次验证：
- `http://127.0.0.1:8765/` 返回 `302 FOUND`

说明：
- Flask 服务没有挂；
- 后端依然在本机运行；
- 当前问题只剩“ESP32 如何访问到这台 Mac”。

### 3. 命令行还没有拿到手机热点网段 IP
本轮命令行仍然没有看到类似：
- `172.20.10.x`

所以现在不能贸然把新的 `host:port` 改成一个猜测值。

## 当前最可能的解释

### 情况 A：Mac 还没有真正完成切到手机热点
UI 里看上去像选中了热点，但系统底层接口还没有真正完成切换。

### 情况 B：当前热点连接走的不是本轮命令检查到的那个接口
也就是：
- 不是 `en0`
- 或者当前系统状态还没有完全刷新

### 情况 C：截图是刚切换的瞬间，命令行检查时状态还没稳定
这种情况在 macOS 上偶尔会出现短时不同步。

## 现在应该怎么继续

### 第 1 步：先不要改代码
当前不是代码问题，不要修改 `.ino`。

### 第 2 步：强制确认 Mac 是否真的联网到手机热点
让用户在 Mac 上做一个最直接的动作：
- 打开浏览器访问一个公网网址，例如：
  - `http://neverssl.com/`
  - 或 `https://example.com/`

如果能打开，说明：
- Mac 确实已经通过热点联网。

### 第 3 步：在终端手动再查一次当前 IP
执行：
- `ipconfig getifaddr en0`
- 如果为空，再执行：
- `ipconfig getifaddr en1`

若仍不对，再执行：
- `ifconfig | grep 'inet '`

目标是找到一个新的热点网段 IP，通常应接近：
- `172.20.10.x`

### 第 4 步：只有拿到这个新 IP，才能回写到 ESP32 的 host:port
正确填写格式应为：
- `<Mac热点IP>:8765`

例如：
- `172.20.10.2:8765`
- 或 `172.20.10.3:8765`

## 当前推进位置

截至本轮，项目推进位置更新为：
- `ESP32 已连上手机热点并拿到 172.20.10.10`
- `Mac UI 看起来也切到了手机热点`
- `但命令行尚未确认 Mac 的热点网段 IP`
- `当前下一步必须先拿到 Mac 在热点下的新 IP，再修改 ESP32 的服务器地址`
