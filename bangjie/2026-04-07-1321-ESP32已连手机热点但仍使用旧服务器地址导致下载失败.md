# ESP32已连手机热点但仍使用旧服务器地址导致下载失败

记录时间：2026-04-07 13:21（Asia/Shanghai）

## 本轮结论

这一步已经再次向前推进了很多：
- **ESP32 已成功连上新的手机热点；**
- **时间同步也成功了；**
- **程序已经开始尝试下载墨水屏 BIN；**
- **但下载地址仍然是旧的服务器地址 `10.142.16.119:8765`，所以最终下载失败。**

也就是说，现在的问题已经不是：
- 烧录问题；
- 串口问题；
- AP 配网页问题；
- 目标 Wi‑Fi 连接问题；

而是一个更具体的配置问题：
- **`host:port` 还没有切换成当前手机热点网络下可达的那台 Mac 地址。**

用户截图串口中已经出现：

这串日志已经可以完整说明流程：
1. ESP32 已经成功连上手机热点 `mengbj_iPhone`；
2. 还拿到了一个热点网段内地址 `172.20.10.10`；
3. NTP 时间同步也成功，说明联网能力是正常的；
4. 随后它去下载 BIN 文件；
5. 但请求的服务器地址仍然是旧的 `10.142.16.119:8765`；
6. 在当前手机热点网络下，这个旧地址不可达，所以 `HTTP code=-1`；
7. 最后下载失败并进入休眠。

## 为什么会这样

因为前面在配网页中填写的：
- `服务器(host:port)`
仍然是之前办公室/原局域网使用的地址：
- `10.142.16.119:8765`

但现在 ESP32 已经不在原来的局域网了，而是在：
- 手机热点网络（172.20.10.x 段）

如果 Mac 没有同时切到这个手机热点，或者虽然切到了但 `host:port` 没更新，ESP32 就找不到服务器。

## 本轮在本机侧的补充检查

本轮再次检查到：
- 当前本机 `8765` 端口上的 Flask 服务仍然在监听；
- 但命令行看到的本机 IP 仍然是：`10.142.16.119`

这进一步说明至少当前这边看到的 Mac 网络还没有切换到和 ESP32 一样的 `172.20.10.x` 热点网段。

因此当前最合理的判断是：
- **ESP32 已切到手机热点；**
- **但 Mac 侧还没有切到同一热点，或者配网页里保留了旧 host:port。**

## 现在应该怎么继续

### 第 1 步：让 Mac 也连接到同一个手机热点
必须让这台跑 `server.py` 的 Mac 也连接到：
- `mengbj_iPhone`

否则：
- ESP32 和 Mac 不在同一张网里；
- `10.142.16.119:8765` 也不可能从手机热点网段访问到。

### 第 2 步：重新确认 Mac 在手机热点下的新 IP
Mac 连接到手机热点后，会拿到一个新的地址，通常类似：
- `172.20.10.x`

这个地址才是你应该填写到 ESP32 配网页里的：
- `服务器(host:port) = <新的172.20.10.x>:8765`

### 第 3 步：重新进入 ESP32 配网页，改掉旧 host:port
因为当前板子已经下载失败并休眠，接下来要：
1. 按一下 `RST`；
2. 重新连接 `InkTime-090`；
3. 打开 `http://192.168.4.1/`；
4. Wi‑Fi 仍填 `mengbj_iPhone`；
5. `服务器(host:port)` 改成 Mac 在手机热点下的新 IP；
6. 再次点击 `保存并重启`。

## 成功后下一次串口应该看到什么

如果这次 host:port 改对了，下一次串口更理想的表现应当是：
- `[WIFI] connected`
- `[WIFI] IP=172.20.10.x`
- `[TIME] OK: ...`
- `[HTTP] GET http://172.20.10.x:8765/static/inktime/yourdownloadkey/photo_N.bin`

### 情况 A：如果服务端静态文件还没准备好
你会看到：
- HTTP 404 或其他非 200 返回

则会继续看到：
- `content-length=...`
- `total read=...`
- 后续才有机会真正刷新墨水屏。
