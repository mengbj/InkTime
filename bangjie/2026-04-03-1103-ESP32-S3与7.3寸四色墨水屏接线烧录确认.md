# ESP32-S3 与 7.3 寸四色墨水屏接线烧录确认

## 这次推进后的结论

- 仓库里的 `esp32/ink-display-7C-photo/ink-display-7C-photo.ino` 和你现在手上的硬件是能对上的，不需要先改 GPIO 定义。
- 我把仓库里的 JLCEDA PCB 工程反向核对了一遍，确认项目作者原始自制板的 8Pin 墨水屏接口，实际就是把 ESP32-S3 的 `GPIO9 ~ GPIO14` 依次接到了转接板的 `DIN/SCLK/CS/DC/RST/BUSY`。
- 这和当前固件里的默认引脚完全一致：

```cpp
#define PIN_EPD_BUSY 14
#define PIN_EPD_RST  13
#define PIN_EPD_DC   12
#define PIN_EPD_CS   11
#define PIN_EPD_SCLK 10
#define PIN_EPD_DIN  9
```

- 也就是说：如果你现在用面包板 + 这块 ESP32-S3 开发板来替代仓库中的自制主板，最稳妥的方式就是按上面这组 GPIO 直接飞线。
- 7.3 寸四色屏在仓库里使用的是 `GxEPD2_730c_GDEY073D46`，逻辑分辨率为 `800x480`，绘制缓冲区按竖屏使用 `480x800`。
- 固件里会优先把 `480 x 800 = 384000` 字节 framebuffer 分配到 PSRAM；因此 Arduino IDE 里 `PSRAM` 必须开启，否则就容易出现分配失败、下载失败、显示失败或运行不稳定。

## 已核对到的硬件对应关系

### 1. 你的转接板接口定义

从你给的转接板照片上，可以直接读到 8Pin 接口丝印，顺序是：

- `BUSY`
- `RST`
- `DC`
- `CS`
- `SCL`
- `SDA`
- `GND`
- `3V3`（板上常见写法可能是 `VCC/3V3`，你这块板照片上电源丝印不完整，但从仓库 PCB 反查可确定这里应为 3.3V 电源）

这里要注意：

- `SCL` 对应 SPI 时钟，也就是 `SCLK`
- `SDA` 在这块板上不是 I2C，而是 SPI 数据输入，也就是 `DIN/MOSI`

### 2. 仓库自制主板的原始定义

我从 `esp32/pcb/InkTime_JLC_EDA.zip` 里的 PCB/封装数据反查了作者自制主板的 8Pin 墨水屏口，得到的实际映射如下：

| 转接板信号 | 仓库固件 GPIO | 说明 |
| --- | --- | --- |
| BUSY | GPIO14 | 忙信号输入 |
| RST | GPIO13 | 墨水屏复位 |
| DC | GPIO12 | 数据/命令选择 |
| CS | GPIO11 | SPI 片选 |
| SCL / SCLK | GPIO10 | SPI 时钟 |
| SDA / DIN / MOSI | GPIO9 | SPI 数据输出 |
| GND | GND | 地 |
| 3V3 | 3V3 | 3.3V 供电 |

这和当前 `.ino` 里的定义完全一致，所以结论非常明确：

> 你现在不用先改固件 GPIO，先按上表接线即可。

### 3. 你手上的 ESP32-S3 开发板如何接

你照片里的开发板不是仓库中那块定制 PCB，而是一块通用 `ESP32-S3` 开发板。实际接法建议如下：

| 转接板 | ESP32-S3 开发板 |
| --- | --- |
| BUSY | `GPIO14` |
| RST | `GPIO13` |
| DC | `GPIO12` |
| CS | `GPIO11` |
| SCL | `GPIO10` |
| SDA | `GPIO9` |
| GND | `GND` |
| 3V3 | `3V3` |

### 4. 接线时的两个关键提醒

- 一定是 **3.3V**，不要接 5V。
- 这块转接板上的 `SDA/SCL` 是 SPI 命名习惯，不是 I2C，总线要按 SPI 接。

## 固件侧确认结果

### 1. 驱动型号已匹配 7.3 寸四色屏

固件当前使用：

```cpp
GxEPD2_7C<
  GxEPD2_730c_GDEY073D46,
  GxEPD2_730c_GDEY073D46::HEIGHT / 4
> display(...)
```

这说明当前固件明确就是给 `GDEY073D46` 这类 7.3 寸四色/7-color 面板准备的，不是误用 13.3 寸或黑白屏驱动。

### 2. PSRAM 不是可选项，而是必选项

固件下载图片时会申请：

```cpp
size_t target = (size_t)FB_WIDTH * FB_HEIGHT; // 384000 bytes
framebuffer = (uint8_t*)heap_caps_malloc(
  target,
  MALLOC_CAP_8BIT | MALLOC_CAP_SPIRAM
);
```

如果 PSRAM 没开，它会退回内部 RAM；但对 ESP32-S3 这种还要跑 WiFi、HTTP、Web 配网、时间同步、GxEPD2 绘制的场景，内部 RAM 裕量会很紧。

因此在 Arduino IDE 里：

- `Tools -> PSRAM` 设为 `Enabled`
- 如果你的核心版本显示为模式项，则优先选和模块一致的 `OPI PSRAM` / `R8` 组合（你仓库 PCB BOM 使用的是 `ESP32-S3-WROOM-1-N8R8`，这是 8MB Flash + 8MB PSRAM 的典型组合）

### 3. 首次联网配置逻辑没有问题

固件行为如下：

- 第一次没有保存 WiFi 时，进入 AP 配网页
- 热点名形如 `InkTime-xxxx`
- 配置页地址为 `http://192.168.4.1/`
- 保存后重启，进入联网下载与刷新逻辑

这和之前分析一致。

### 4. 恢复出厂设置逻辑

固件定义了：

```cpp
#define PIN_FACTORY_RESET 38
```

即：上电时若 `GPIO38` 被拉低，就清除保存的 WiFi 和服务器配置。

对你现在这块通用 ESP32-S3 板子来说：

- 如果没有额外接按钮，也没关系，程序照常运行
- 以后如果要做“清空配网”按钮，可以把 `GPIO38` 接一个按键到 GND

### 1. Python 代码语法检查已通过


结果通过，没有语法错误。

### 2. 固件下载路径规则没有问题

固件默认下载前缀还是：

```cpp
#define DAILY_PHOTO_PATH_PREFIX "/static/inktime/yourdownloadkey/photo_"
```

所以你真正使用前，必须把它和 `config.py` 里的：

```python
DOWNLOAD_KEY = "yourdownloadkey"
```

同步改成同一个值。

例如如果你改成：

```python
DOWNLOAD_KEY = "abc123xyz"
```

那固件也必须改为：

```cpp
#define DAILY_PHOTO_PATH_PREFIX "/static/inktime/abc123xyz/photo_"
```

否则 ESP32 连接服务器时就会拿到 404。

### 3. 服务器地址怎么填

你前面给出的说明是正确的：

- 要填你电脑在局域网里的地址，例如 `192.168.1.20:8765`
- 不要填 `127.0.0.1:8765`

原因是：

- `127.0.0.1` 对 ESP32 来说，只表示“ESP32 自己”
- 它不会指向你电脑上的 Flask 服务

## 如果现场报错，优先这样排查

## 1. 烧录失败 / 一直连不上板子

优先排查：

- 数据线是否是可传输数据的 USB 线，而不是纯供电线
- Arduino IDE 中选择的串口是否正确
- 是否需要手动按 `BOOT`
- 上传速度是否过高
- 是否误选成了非 S3 板型

## 2. 能联网但下载失败

看串口中的 HTTP 返回码：

- `404`：大概率是 `DOWNLOAD_KEY` 不一致
- `-1`：大概率是服务器地址写错、电脑防火墙拦截、ESP32 与电脑不在同一局域网
- `size mismatch`：服务端返回内容不是完整的 `480x800` 单字节 bin 文件

## 3. 下载成功但屏幕不刷新

优先排查：

- `BUSY/RST/DC/CS/SCLK/DIN` 有没有接错一根
- 供电是否稳定在 3.3V
- FPC 排线方向是否装反
- 转接板和屏幕是否确实是一套兼容面板

## 4. 刷新异常或随机死机

优先排查：

- `PSRAM` 是否真的开启
- Flash / PSRAM 选项是否和模块型号匹配
- USB 供电是否不足
- 面包板接触是否松动

## 关于模型，现在需要什么样的模型

服务端 `analyze_photos.py` 会把图片以 OpenAI 兼容 `chat/completions` 多模态请求发给 VLM，因此你需要的是：

- **支持图像输入的视觉语言模型（VLM）**
- **接口兼容 OpenAI `chat/completions`**
- **最好能本地部署，或至少在局域网内稳定访问**

### 可直接用的模型类型

优先推荐两档：

1. 能力和速度较平衡：
   - `qwen2.5-vl-7b-instruct`
   - `qwen2.5-vl-32b-instruct`
2. 如果你已经验证过兼容：
   - `qwen3-vl-32b-instruct`

### 什么时候需要换模型

如果你发现：

- 对照片地点/事件理解差
- 标题和摘要经常跑偏
- 人物/场景识别不稳定

那就应该优先升级到更强的 VLM，而不是先折腾墨水屏代码。

### 当前最务实建议

如果你只是先把链路跑通：

- 先用一个能稳定返回 JSON 的本地 VLM
- 不用一开始就追求最强模型
- 先确认 `analyze -> render -> server -> esp32 -> epd` 这条链路完整跑通
