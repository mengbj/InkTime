# Mac未连手机热点导致HTTP code=-1

记录时间：2026-04-07 14:08（Asia/Shanghai）

## 本轮结论

这一轮已经把问题定位得非常清楚了：

- ESP32 重新配网后，**已经成功连上手机热点 `mengbj_iPhone`**
- 也已经拿到了热点网段 IP：`172.20.10.10`
- 时间同步也成功：`[TIME] OK: 2026-04-07 14:08:03`
- 但 HTTP 阶段仍然是：`[HTTP] code=-1`

根因不是 ESP32，也不是配网页保存失败，而是：

**此刻 Mac 自己已经不在手机热点网络里了，所以 `172.20.10.9:8765` 对 ESP32 来说不可达。**

这两个信息合在一起，就说明：

- ESP32 在手机热点网段 `172.20.10.x`

- ESP32 访问 `172.20.10.9:8765` 时，自然找不到 Mac 上的服务

因此才会出现：

- `HTTP code=-1`

这通常就是 TCP 连接不上，而不是文件 404。

## 现在最应该怎么做

### 方案 A：继续用手机热点测试（推荐当前先这样）

你现在要做的是把 **Mac 重新连回 `mengbj_iPhone`**。

操作顺序：

1. 点击 Mac 右上角 Wi-Fi
2. 手动切回：`mengbj_iPhone`
3. 终端执行：

```bash
ipconfig getifaddr en0
```

4. 记下新的热点 IP

如果重新回到手机热点，通常会再次拿到类似：

- `172.20.10.9`

也可能不是 `.9`，例如 `.8`、`.11`，这都正常，**以实际命令输出为准**。

5. 如果 IP 变了，就重新进 ESP32 配网页 `http://192.168.4.1/`，把服务器地址改成：

```text
新的热点IP:8765
```

例如：

```text
172.20.10.9:8765
```

或

```text
172.20.10.11:8765
```

6. 保存后再看串口，目标应变成：

- `[WIFI] connected`
- `[TIME] OK`
- `HTTP code=200`
- `Content-Length=384000`

---

### 方案 B：改成同一路由器网络测试

如果你不想依赖手机热点反复切换，也可以让：

- Mac 连 `JYXC-Renzheng`
- ESP32 也去连 `JYXC-Renzheng`

然后把服务器地址改成 Mac 在这个路由器下的局域网 IP，例如现在已经看到的是：

- `10.142.16.119`

前提是：

- ESP32 能成功连接这个 Wi-Fi
- Mac 防火墙没有拦截 `8765`
- 路由器内网设备之间允许互访

当前为了减少变量，**建议先继续用方案 A，把手机热点链路打通后先点亮屏幕。**
